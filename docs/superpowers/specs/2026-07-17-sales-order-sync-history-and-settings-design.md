# 销量订单同步记录与全局设置设计

## 目标

为商品销量分析提供独立的订单获取历史和全局调度设置：

1. 不把销量订单同步记录写入现有 `lt_sync_tasks`，避免与商品、上下架、删除和替换同步任务混合。
2. 在“AI 管理”下新增“订单获取记录”，按店铺展示每次自动或手动订单获取的执行历史。
3. 在“商品分析设置”新增“订单同步设置”标签页，仅超级管理员可见并可修改全系统统一的自动同步周期、启用状态和成功记录保留天数。

## 导航结构

```text
AI 管理
├─ 标题优化
├─ 商品分析
├─ 商品分析设置
└─ 订单获取记录
```

新增路由：

```text
/ai/order-sync-history
```

路由要求 `ai.manage` 权限。

“商品分析设置”新增第四个标签页：

```text
订单同步设置
```

该标签页只有超级管理员可见。普通用户既看不到标签，也不能调用修改接口。

## 全局订单同步设置

配置是全系统统一值，不按用户或店铺分别保存。

字段：

| 字段 | 默认值 | 允许范围 | 说明 |
|---|---:|---:|---|
| 启用自动同步 | 开启 | 开启或关闭 | 关闭后停止自动订单同步，手动“立即更新”仍可使用 |
| 自动同步间隔 | 30 分钟 | 5 至 1440 分钟 | 每家店铺以上次成功同步时间为基准判断是否到期 |
| 成功记录保留天数 | 30 天 | 1 至 365 天 | 超期的成功记录自动删除 |

设置保存在现有系统设置表中，使用独立设置键，例如：

```text
sales_order_sync_settings
```

公共结构：

```json
{
  "enabled": true,
  "intervalMinutes": 30,
  "successRetentionDays": 30
}
```

保存规则：

- 只有点击“保存设置”才持久化。
- 页面加载和切换标签不得自动保存。
- “恢复默认”只修改当前草稿，仍需点击保存。
- 后端 PUT 接口只允许超级管理员调用。
- 设置保存后无需重启服务，从下一次调度检查开始生效。

## 自动调度

后台调度线程仍每约 60 秒检查一次到期店铺。

调度规则：

- 全局 `enabled=false` 时不创建自动任务。
- 仅处理启用店铺。
- 仅处理具有完整乐天订单 API 凭据的店铺。
- 店铺已有有效订单同步任务时不重复创建。
- 以上次成功同步时间和全局 `intervalMinutes` 判断是否到期。
- 首次同步仍默认获取最近 90 天。
- 后续执行增量同步和近期订单复查。
- 修改间隔后，下一次检查立即使用新值。

## 独立历史数据模型

新增表：

```text
lt_sales_order_sync_runs
```

字段：

- `id`：字符串主键。
- `owner_username`：店铺所有者。
- `store_id`：店铺 ID，店铺删除后允许置空。
- `store_name`：执行时店铺名称快照。
- `trigger_type`：`automatic`、`manual` 或 `retry`。
- `parent_run_id`：重试来源记录，可空。
- `status`：`queued`、`running`、`success`、`partial`、`failed` 或 `cancelled`。
- `initial_sync`：是否为首次 90 天同步。
- `progress_current`。
- `progress_total`。
- `total_order_count`：本次从乐天检查的订单数量。
- `new_order_count`：新增订单数量。
- `updated_order_count`：更新已有订单数量。
- `unchanged_order_count`：无变化订单数量。
- `failed_order_count`。
- `message`：当前进度或完成摘要。
- `error_detail`：脱敏错误信息。
- `started_at`。
- `finished_at`。
- `created_at`。
- `updated_at`。

索引：

- `owner_username + created_at`
- `owner_username + status`
- `store_id + created_at`
- `status + finished_at`

历史记录是每次执行一条，不覆盖上一条。

现有 `lt_sales_sync_states` 继续保存每家店铺的最新同步状态、游标和页面进度。两张表职责区分：

- `lt_sales_sync_states`：每家店铺当前和最近状态。
- `lt_sales_order_sync_runs`：每次执行的历史记录。

## 触发方式

### 自动

定时调度到期时：

1. 创建 `trigger_type=automatic` 的历史记录。
2. 状态设为 `queued`。
3. 把历史记录 ID 传入销量同步 Worker。
4. Worker 更新该记录的进度和最终结果。

### 手动

用户在商品分析页面点击“立即更新”时：

1. 创建 `trigger_type=manual` 的历史记录。
2. 状态设为 `queued`。
3. 商品分析页面继续读取现有店铺同步状态。
4. 订单获取记录页面可以看到本次手动执行。

### 重试

失败、部分成功或已终止记录可以重试：

1. 不修改原记录。
2. 创建新的 `trigger_type=retry` 记录。
3. 新记录的 `parent_run_id` 指向原记录。
4. 同一个店铺已有有效任务时拒绝重复重试。

## 执行状态与进度

状态流转：

```text
queued → running → success
                  → partial
                  → failed
                  → cancelled
```

执行过程中：

- 同步状态表和历史记录同时更新进度。
- 页面轮询运行中的记录。
- Worker 异常退出时，通过超时恢复逻辑把遗留的 `queued/running` 记录转为 `failed`。
- 错误信息必须脱敏，不保存店铺密钥、AI Key、数据库连接信息或第三方完整响应。

## 结果计数

同步服务返回统一结果：

```json
{
  "totalOrderCount": 120,
  "newOrderCount": 8,
  "updatedOrderCount": 17,
  "unchangedOrderCount": 94,
  "failedOrderCount": 1
}
```

状态判定：

- `failedOrderCount=0`：`success`
- 有成功处理且 `failedOrderCount>0`：`partial`
- 全部失败或执行级异常：`failed`

## 订单获取记录页面

页面名称：

```text
订单获取记录
```

页面顶部显示：

- 自动同步当前是否启用。
- 当前全局同步间隔。
- 成功记录保留天数。
- 最近刷新时间。

普通用户可以查看这些全局值，但不能编辑。

列表筛选：

- 店铺。
- 触发方式。
- 状态。
- 创建时间范围。

表格字段：

- 任务名称。
- 店铺。
- 触发方式。
- 状态。
- 是否首次同步。
- 订单总数。
- 新增订单。
- 更新订单。
- 无变化订单。
- 失败订单。
- 进度。
- 错误信息。
- 创建时间。
- 开始时间。
- 完成时间。
- 耗时。
- 操作。

任务名称格式：

```text
销量订单同步（自动）- 店铺名称
销量订单同步（手动）- 店铺名称
销量订单同步（重试）- 店铺名称
```

操作：

- `queued/running`：仅展示状态，不提供前端强制终止，避免正在写入订单事实表时产生不完整中断。
- `failed/partial/cancelled`：显示“重试”。
- 支持选择记录后批量删除自己的历史记录，但不能删除运行中的记录。
- 删除历史记录不删除订单、销量数据或店铺同步状态。

分页：

- 使用服务器分页。
- 默认每页 30 条。
- 运行中记录每 2 秒刷新当前页。

## 权限与隔离

订单获取记录：

- 用户只能查看自己名下店铺的记录。
- 用户只能重试和删除自己的记录。
- 超级管理员不因角色自动获得其他用户记录。
- 所有后端查询使用 `owner_username` 强制过滤。

全局同步设置：

- GET：拥有 `ai.manage` 的用户可读取，用于显示当前周期。
- PUT：仅超级管理员可修改。
- 前端仅超级管理员显示“订单同步设置”标签。
- 后端不能依赖前端隐藏，必须强制超级管理员校验。

## API

全局设置：

```text
GET /crawler/settings/sales-order-sync
PUT /crawler/settings/sales-order-sync
```

历史记录：

```text
GET /crawler/sales-analysis/order-sync-runs
POST /crawler/sales-analysis/order-sync-runs/{run_id}/retry
DELETE /crawler/sales-analysis/order-sync-runs
```

列表参数：

- `page`
- `pageSize`
- `storeId`
- `triggerType`
- `status`
- `createdAtFrom`
- `createdAtTo`

删除请求：

```json
{
  "runIds": ["..."]
}
```

## 自动清理

调度线程定期清理历史：

- 只删除 `status=success`。
- `finished_at` 早于当前时间减去 `successRetentionDays`。
- 不删除 `queued`、`running`、`partial`、`failed` 或 `cancelled`。
- 自动和手动成功记录使用同一个保留天数。
- 清理过程设置进程内锁，避免重复执行。
- 清理失败只记录日志，不阻断订单同步调度。

## 前端商品分析联动

商品分析页面保持现有体验：

- “立即更新”按钮不变。
- 店铺最新同步状态和进度不变。
- 每次手动更新同时生成一条历史记录。
- 页面不需要跳转到普通“同步任务”。

可在同步状态区域增加“查看订单获取记录”入口，跳转到新页面并带当前店铺筛选参数。

## 兼容与迁移

- 新表通过现有数据库初始化兼容流程创建。
- 全局设置不存在时使用默认值，不在 GET 时自动写入。
- 第一次由超级管理员保存时才持久化设置。
- 现有 `lt_sales_sync_states` 数据保持不变。
- 上线前已有同步没有历史记录，不进行伪造回填。
- 新功能上线后的每次自动和手动同步开始生成记录。
- MySQL DDL 必须覆盖生产版本兼容测试，TEXT/LONGTEXT 列不得使用不兼容的数据库默认值。

## 测试要求

### 后端

- 默认全局设置返回 30 分钟、启用和保留 30 天，GET 不写库。
- 非超级管理员 PUT 返回 403。
- 设置范围校验正确。
- 自动调度使用数据库设置而不是固定常量。
- 关闭自动同步后不创建自动记录。
- 手动同步仍可执行并创建记录。
- 自动、手动和重试记录触发类型正确。
- 同店铺有效任务去重。
- 进度和结果计数正确写入。
- 所有列表、重试和删除按 owner 隔离。
- 运行中记录不能删除。
- 成功记录按可配置天数清理，失败记录保留。
- MySQL 新表和兼容列 DDL 可执行。

### 前端

- AI 管理出现“订单获取记录”。
- 路由受 `ai.manage` 保护。
- 设置页“订单同步设置”仅超级管理员可见。
- 全局设置只有点击保存才持久化。
- 普通用户可以在记录页查看周期但不能编辑。
- 列表筛选、分页、轮询、重试和删除行为正确。
- 任务类型、触发方式、状态和计数文案正确。
- 商品分析页可以跳转到当前店铺记录。
- 生产构建通过。

## 验收标准

1. 销量订单同步不写入普通 `lt_sync_tasks`。
2. 每次自动、手动和重试同步生成独立订单获取记录。
3. 用户只能查看和操作自己的记录。
4. 超级管理员可以在商品分析设置中配置全局启用状态、同步间隔和成功记录保留天数。
5. 普通用户不能看到或调用修改功能。
6. 成功记录按配置自动清理，非成功记录保留。
7. 商品分析页面最新同步状态与历史记录保持一致。
