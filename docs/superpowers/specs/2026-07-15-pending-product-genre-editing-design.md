# 待审核商品品类展示、修改与审核校验设计

## 目标

在待审核商品列表中显示乐天品类编号和完整品类路径，允许通过可搜索下拉框修改品类。商品没有有效品类编号及对应路径时，不允许单条或批量审核通过。

## 范围

- 待审核商品列表展示品类编号和完整路径。
- 提供品类搜索接口，支持按 6 位编号或完整路径关键词查询。
- 提供待审核商品品类修改接口。
- 前端在单条和批量审核前提示无有效品类的商品。
- 后端在审核状态变更接口中强制执行相同校验。
- 不修改已审核、异常、上架中或已上架商品的品类编辑行为。

## 数据来源

品类数据继续使用后端现有的 `app/resources/rakuten_attribute_rules.json`。

- `genres` 对象的键作为品类编号。
- 每个品类的 `genrePath` 作为完整品类路径。
- 商品表继续以 `genre_id` 作为持久化字段，不新增重复的品类名称字段。
- 商品对外结构增加根据当前 `genre_id` 动态解析的 `genrePath`。

这样可以保证品类名称始终与编号对应，避免数据库中的编号和文本分别修改后产生不一致。

## 后端设计

### 品类查询

新增需要商品管理权限的品类搜索接口。

- 接收关键词和结果数量限制。
- 纯数字关键词同时匹配品类编号。
- 普通文本匹配完整 `genrePath`。
- 返回 `genreId` 和 `genrePath`。
- 默认限制返回数量，避免一次传输约 1.4 万条品类。
- 无关键词时可返回当前品类，或返回有限的初始结果，不加载全部数据。

### 品类修改

新增单商品品类修改接口。

- 商品必须属于当前用户。
- 商品状态必须是 `pending`。
- `genreId` 必须是 6 位数字。
- `genreId` 必须存在于本地品类数据中。
- 更新 `ProductModel.genre_id`。
- 同步更新商品原始载荷中的 `genreId`，确保详情、预检和后续上架读取一致。
- 返回更新后的商品公开结构。

### 商品公开结构

列表和详情的商品结构增加：

- `genreId`
- `genrePath`

当编号为空或无法匹配本地品类数据时，`genrePath` 返回空字符串。

### 审核校验

在 `update_product_status` 将商品改为 `approved` 前校验本次操作的全部商品。

- 品类编号不能为空。
- 品类编号必须是 6 位数字。
- 品类编号必须存在于本地品类数据中，并能解析出非空完整路径。
- 任一商品不满足条件时，整批操作失败，不修改任何商品状态。
- 错误信息包含无效商品数量及有限数量的商品名称，便于定位。

该校验在事务内、状态写入前完成，确保批量审核不会部分成功。

## 前端设计

### 类型和 API

- `ProductItem` 与详情类型增加 `genrePath`。
- 增加 `RakutenGenreOption` 类型。
- API 组合函数增加搜索品类和修改商品品类的方法。

### 列表展示

仅在待审核列表的商品信息区域显示品类编辑控件。

- 下拉框当前值显示完整品类路径。
- 下方或辅助文本显示 `品类编号：xxxxxx`。
- 无品类时显示明确的空状态，例如“请选择品类”与“品类编号：未设置”。
- 下拉框支持远程搜索，可输入品类编号或完整路径关键词。
- 搜索结果显示完整路径和编号。

品类控件放入现有商品列表项，不新建页面或弹窗。

### 修改交互

- 用户选中下拉项后立即调用品类修改接口。
- 保存期间禁用品类控件，防止重复提交。
- 保存成功后用接口返回值替换当前列表商品。
- 保存失败时恢复原选择并显示接口错误。
- 该列表内联修改是独立的即时保存行为，不改变商品详情弹窗只有点击“保存”才持久化的规则。

### 审核提示

单条审核：

- 确认弹窗前检查当前商品是否同时具有有效 `genreId` 和 `genrePath`。
- 不满足时不打开确认弹窗，提示“请先选择有效品类，当前商品不能审核通过”。

批量审核：

- 检查所有已选择商品。
- 有任意无效商品时整批不提交。
- 提示无效数量，并显示有限数量的商品名称。

前端校验用于及时反馈，后端校验是最终约束。

## 组件边界

当前功能继续由 `ProductWorkflowView.vue` 编排列表数据和审核操作。

新增一个聚焦的品类选择组件：

- 输入：商品 ID、当前品类编号、当前完整路径、禁用状态。
- 输出：保存成功后的商品或品类变更事件。
- 内部负责远程搜索、选项加载、保存中状态和失败回退。

视图组件负责用保存结果更新列表数据，并负责单条、批量审核前校验。后端负责数据正确性和权限边界。

## 错误处理

- 品类搜索失败：保留当前品类显示，并提示加载失败。
- 品类保存失败：恢复原值，显示后端错误。
- 商品已不再是待审核状态：后端拒绝修改，前端刷新列表。
- 品类编号不存在：后端拒绝保存和审核。
- 批量审核存在无效品类：整批失败，不产生部分状态变更。

## 测试与校验

后端自动化测试：

- 按编号和路径搜索品类。
- 修改待审核商品为有效品类。
- 拒绝无效、未知品类编号。
- 拒绝修改非待审核商品。
- 单条无品类商品不能审核通过。
- 批量商品中存在无品类商品时全部不能审核通过。
- 有效品类商品可以正常审核。

前端自动化或可测试纯逻辑：

- 有效品类判断。
- 单条审核缺少品类时阻止请求。
- 批量审核列出无效商品并阻止请求。
- 品类选择成功后更新编号和路径。
- 保存失败时恢复原值。

验证只运行现有测试、类型检查和构建检查，不启动前端或后端应用，不打开界面检查。

## 非目标

- 不新增独立品类管理页面。
- 不允许自由输入不存在于本地规则文件的品类。
- 不批量修复历史商品品类。
- 不修改详情弹窗的显式保存语义。
- 不启动或重新运行应用进行界面验证。
